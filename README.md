# Coding Style Prompts

一个包含多种编程语言工程级编码规范的提示词集合。这些规范旨在帮助生成可维护、结构清晰、符合工程实践的高质量代码。

## 📁 项目结构

```
coding_style_prompt/
├── README.md              # 项目说明文档
├── .gitignore             # Git 忽略文件
└── styles/                # 编码规范文件目录
    ├── c++_style.md       # C++ 编码规范
    ├── python_style.md    # Python 编码规范
    └── javascript_style.md # JavaScript 编码规范
```

## 📋 包含的语言规范

### C++ 编码规范 (`c++_style.md`)

针对系统软件工程师的工程级 C++ 编码规范，包含：

- **整体原则**：代码可读性、控制流清晰、类型显式
- **头文件规范**：禁止使用 `<bits/stdc++.h>`，必须显式引入
- **命名规范**：camelCase 变量、动词开头函数、PascalCase 类
- **循环规范**：优先使用 `size_t` 索引或范围 for 循环
- **auto 使用规范**：类型必须明确可读
- **Lambda 使用规范**：仅用于简单回调和比较函数
- **数据结构规范**：优先使用 struct 代替 pair
- **内存管理**：使用智能指针，禁止手动 new/delete
- **错误处理**：必须使用异常机制
- **代码结构**：项目分层设计（include/src/service/model/utils）

### Python 编码规范 (`python_style.md`)

针对工程架构师的 Python 编码规范，包含：

- **命名规范**：PascalCase 类、snake_case 函数/变量、UPPER_CASE 常量
- **类型系统**：必须使用类型注解，优先 Python 3.10+ 语法
- **代码结构**：分层设计（core/service/model/utils/config）
- **循环与数据处理**：优先显式循环，禁止复杂嵌套推导式
- **Lambda 使用**：禁止滥用，仅用于简单回调
- **异常处理**：中英双语错误信息
- **日志规范**：使用 logging 模块，禁止 print
- **魔法数字**：必须提取为常量
- **字符串处理**：优先使用 f-string
- **导入规范**：标准库、第三方库、相对导入分区

### JavaScript 编码规范 (`javascript_style.md`)

针对后端工程师的 JavaScript 编码规范，包含：

- **整体原则**：清晰度和可维护性优先
- **命名规范**：camelCase 变量/函数、PascalCase 类、snake_case 文件名
- **循环与数据处理**：优先使用显式 for...of 循环
- **函数规范**：优先普通函数，箭头函数仅用于简单回调
- **控制流**：所有条件语句必须使用大括号
- **模块规范**：使用 CommonJS（require/module.exports）
- **错误处理**：统一返回结构 `{success, message, data}`
- **禁止事项**：禁止滥用可选链、复杂三元表达式、超长单行逻辑

## 🚀 使用方法

将对应语言的规范文件内容提供给 AI 助手，即可生成符合工程实践的代码。

示例：

```
[粘贴对应语言的编码规范]

请根据以上规范，编写一个 [具体功能描述] 的代码。
```

## 📝 核心理念

所有规范共同遵循的原则：

1. **可读性优先**：代码应该清晰易懂，不追求语法炫技
2. **类型明确**：避免隐式类型，类型必须可读
3. **结构分层**：项目必须分层设计，禁止单文件包含所有逻辑
4. **禁止滥用**：不滥用语法糖、lambda、函数式链式写法
5. **错误处理**：必须有完善的错误处理机制
